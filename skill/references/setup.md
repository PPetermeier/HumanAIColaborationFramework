## Setup Overview

The framework consists of seven components organized into three layers:

- **Foundation Layer**: Project Foundations, Collaborator Profile, Engagement Guidelines
- **Dynamic Layer**: Living Questions, Project Plan
- **Operational Layer**: Session Template, Working Files

**Recommended setup order**: Project Foundations first (establishes scope and vocabulary), then Collaborator Profile (establishes the human's working context), then remaining components in any order. Each component builds on information gathered in previous ones.

**Setup scope**: Each component can be completed in a single session or split across multiple sessions. When splitting, complete logical sections together, summarize progress at session boundaries, and review prior content when resuming.

**Overall approach**: Use conversational exploration to gather information, then populate the corresponding template. Start broad, narrow to specifics. Validate understanding before documenting. Prioritize practical, actionable content over theoretical completeness.

## Shared Patterns

These patterns apply across all component setups. Individual sections reference them rather than repeating them.

### Exploration Sequence

For each component, follow this sequence:

1. **Orient** -- Explain the component's purpose and what the human should expect (time, structure, output).
2. **Explore broadly** -- Ask open-ended questions to surface the human's context, needs, and preferences.
3. **Refine** -- Follow up on specific areas with targeted questions. Identify patterns across responses.
4. **Document** -- Populate the template section by section. Use direct quotes where they capture intent clearly; summarize patterns otherwise.
5. **Validate** -- Review the populated template with the human. Confirm accuracy, completeness, and relevance.

### Quality Checks

Before finalizing any component:

- Verify all template sections are populated.
- Confirm internal consistency (no contradictions between sections).
- Check that content is specific to this project, not generic placeholder text.
- Validate with the human that the document accurately reflects their intent.
- Identify integration points with other framework components.

### Session Management

When a component setup spans multiple sessions:

- Complete logical subsections together (do not leave a subsection half-populated).
- End each session with a summary of what was covered and what remains.
- Begin the next session by reviewing what was documented previously.
- Allow the human to revise earlier sections as later exploration surfaces new understanding.

## 1. Project Foundations

**Purpose**: Establish the project's scope, vocabulary, assumptions, methods, tooling, and operational structure. This document anchors all other components.

**Template sections to populate**:

- Core Framework (purpose/scope, key definitions, foundational assumptions)
- Intellectual Framework (theoretical foundations, methodological approach, quality standards)
- Technical Infrastructure (knowledge work environment, tool selection rationale, integration architecture, evolution strategy)
- Operational Structure (working patterns, documentation framework, integration mechanisms)
- Evolution Framework (development patterns, learning integration, refinement processes)

### Exploration Guide

**Core Framework** -- Start here. Ask about:
- Overall project goals: "What is this project trying to achieve, and why does it matter?"
- Boundaries: "What is explicitly in scope? What is out of scope?"
- Key terms that need precise definitions for this project context.
- Assumptions the human is working from (surface both explicit and implicit ones). Probe for risk implications and validation needs.

**Intellectual Framework** -- Explore:
- Relevant theoretical frameworks or models. Ask why they were chosen and what their limitations are.
- Methodological approach: what methods will be used, why, and how.
- Quality standards: what "good" looks like for this project. Establish both objective measures and subjective criteria.

**Technical Infrastructure** -- Map the tooling ecosystem:
- Development tools, documentation tools, analysis tools, productivity tools.
- How tools integrate with each other (workflows, data flows).
- Tool selection rationale and evolution strategy.

**Operational Structure** -- Establish:
- Working patterns that fit the human's style.
- Documentation approaches and conventions.
- How different project components integrate.

Use concrete examples to clarify abstract concepts. Document both decisions and their rationale.

## 2. Collaborator Profile

**Purpose**: Document the human's expertise, working style, communication preferences, and project-specific context. Informs how all other components are tailored.

**Template sections to populate**:

- Core Information (technical background, tool proficiency, working style, communication preferences)
- Project Context (project-specific expertise, learning objectives, role understanding)
- Interaction Patterns (session structure preferences, feedback mechanisms, documentation practices)
- Evolution Tracking (initial baseline for profile updates, learning progression, interaction refinements)

### Exploration Guide

Proceed through four areas in sequence:

**Technical Background** -- Start with open questions about experience and expertise. Follow up on project-relevant areas. Explore both strengths and growth areas. Document tools and methodologies they use.

**Working Style** -- Investigate:
- Problem-solving approaches and cognitive preferences.
- Ideal working conditions and energy management.
- How they handle complexity, ambiguity, and context-switching.

**Communication Preferences** -- Determine:
- Preferred interaction style (concise vs. detailed, formal vs. casual).
- Documentation preferences (what format, how much, how often).
- Feedback preferences (how they like to give and receive it).
- Situation-specific adaptations (e.g., different preferences when debugging vs. brainstorming).

**Project Context** -- Connect general profile to this specific project:
- Directly relevant experience and transferable skills.
- Learning objectives for this project.
- Role expectations and responsibilities.

### Documentation Notes

- Use direct quotes where they clearly capture a preference.
- Summarize patterns observed across multiple responses.
- Keep content project-relevant; omit tangential biographical detail.

## 3. Engagement Guidelines

**Purpose**: Define collaboration patterns, session structure, communication framework, quality assurance, and working methods for the human-AI partnership.

**Template sections to populate**:

- Core Engagement Patterns (pattern documentation, session structure, communication framework, session initialization framework)
- Quality Assurance Framework (progress tracking, review mechanisms)
- Working Methods (methodology framework, tool integration)
- Evolution Framework (pattern evolution, progress assessment)
- Documentation Standards (update protocols, quality management)

### Exploration Guide

**Core Patterns** -- Explore through experience-based questions:
- "Describe a particularly effective collaboration experience. What made it work?"
- "What challenges have you encountered in past collaborations?"
- "How do you handle transitions between work sessions?"

From these responses, extract actionable patterns. Each pattern should document: purpose, context where effective, implementation guidance, success indicators, and common pitfalls.

**Session Structure** -- Establish:
- Natural work rhythms and optimal session length.
- Session opening protocols (how to establish context).
- Core work patterns (how to structure productive work time).
- Session closure (how to document and hand off).
- Continuity mechanisms between sessions.

**Session Initialization Framework** -- Define the context transfer protocol:
- Previous session summary format.
- Current position and active project phase.
- Session objectives and success criteria.
- Relevant dependencies and resource status.

**Quality Assurance** -- Develop:
- Progress documentation standards and achievement verification methods.
- Review cycles and their frequency.
- Adjustment triggers (what signals a pattern needs changing).

**Working Methods** -- Document:
- Investigation and analysis patterns.
- Tool usage conventions.
- Documentation standards.

Start with minimal viable patterns. Complexity should be added only as experience reveals the need.

## 4. Living Questions

**Purpose**: Track the project's evolving knowledge state -- what is known, what is uncertain, what is under investigation, and how understanding develops over time.

**Template sections to populate**:

- Current Understanding Framework (core knowledge state, theoretical framework, empirical connection)
- Investigation Framework (active questions, investigation status, research quality)
- Integration Framework (cross-component connections, synthesis development)
- Evolution Framework (understanding development, documentation evolution)

### Exploration Guide

**Current Knowledge State** -- Map the starting landscape:
- "What do you currently know with confidence about this project domain?"
- "What are the major uncertainties in your understanding?"
- "What assumptions underlie your current thinking?"
- "Where do you see knowledge gaps that need investigation?"

Organize responses into: core knowledge components, key uncertainties, critical assumptions, important relationships, and investigation needs.

**Question Development** -- Help the human formulate investigable questions:
1. Start with broad areas of uncertainty.
2. Refine into specific, actionable questions.
3. Categorize: theoretical inquiries, empirical investigations, methodological explorations, integration challenges.
4. For each question, document: current understanding, investigation approach, required resources, expected implications, success indicators.

**Integration** -- Map how knowledge components connect:
- Direct influences and indirect effects between components.
- Feedback loops and resource dependencies.
- How findings in one area affect understanding in others.

**Prioritization** -- Help the human prioritize questions by:
- Impact on project objectives.
- Feasibility of investigation.
- Dependencies (which answers unlock other investigations).

## 5. Project Plan

**Purpose**: Organize the project's work into components, timelines, resource allocations, and progress tracking mechanisms.

**Template sections to populate**:

- Project Overview (context, core objectives, scope definition)
- Project Structure (work components, dependencies/relationships, timeline framework)
- Implementation Framework (resource allocation, quality assurance, risk management)
- Progress Tracking (monitoring framework, reporting structure, adaptation mechanisms)

### Exploration Guide

**Project Overview** -- If Project Foundations has been completed, draw from it. Otherwise, explore:
- "What is the ultimate goal of this project?"
- "What concrete outcomes would signal success?"
- "What constraints or boundaries shape this work?"
- "What resources are available?"
- "What time constraints apply?"

**Work Decomposition** -- Help break the project into components:
- "What are the major pieces of work needed?"
- "How do these pieces relate to each other?"
- "What dependencies exist between components?"
- "How should work be sequenced?"

For each component, document: purpose, success criteria, resource requirements, dependencies, quality standards, and documentation needs.

**Timeline** -- Develop practical project timing:
- Identify natural work phases and logical milestone points.
- Map resource availability against work requirements.
- Establish integration points where components come together.
- Define review cycles and their timing.

**Progress Tracking** -- Establish:
- What indicates meaningful progress (concrete indicators, not vague sentiment).
- How and when progress will be measured.
- What triggers plan adjustments.
- How status will be communicated.

**Risk Management** -- Identify:
- Key risk categories for this project.
- Assessment and mitigation strategies.
- Monitoring protocols.

Balance structure with flexibility. Build in adaptation mechanisms so the plan can evolve without breaking.

## 6. Session Template

**Purpose**: Establish the structure for individual working sessions -- how context is transferred, progress is tracked, knowledge is preserved, and continuity is maintained between sessions.

**Template sections to populate**:

- Session Context (project reference framework, session focus, continuity maintenance)
- Progress Framework (work completed, current status, blocking issues)
- Knowledge Development (understanding evolution, key decisions, open questions)
- Integration Framework (cross-references, modified artifacts, created artifacts)
- Next Steps (immediate actions, future considerations, handoff requirements)

### Exploration Guide

**Work Pattern Assessment** -- Understand how the human works:
- "How do you typically structure your work sessions?"
- "What information do you need to start working effectively?"
- "What helps you maintain context between sessions?"
- "How do you track progress and decisions?"

Organize into: session structure preferences, continuity requirements, progress tracking needs, knowledge transfer approaches.

**Session Structure** -- Define the template elements:
- **Opening**: What context must be established at session start? Define the project reference framework (current status, active objectives, dependencies, resources) and knowledge state reference (relevant Living Questions context).
- **During**: How progress, decisions, and new understanding are captured in real time.
- **Closing**: What must be documented for effective handoff -- work completed, current status, blocking issues, open questions, next steps.

**Integration** -- Establish connections:
- How sessions reference and update the Project Plan.
- How sessions feed into Living Questions (new insights, resolved questions).
- How session documents cross-reference working files and artifacts.

**Implementation** -- Advise starting with a minimal session framework:
1. Use the template in actual sessions.
2. Gather feedback on what works and what is missing.
3. Refine the template based on experience.
4. Document template evolution.

## 7. Working Dialogue

**Purpose**: Establish the organizational structure and documentation standards for the project's working files -- the actual artifacts produced during project work.

**Template output**: Not a single document but a set of requirements and conventions that all working files follow.

### Exploration Guide

**Project Understanding** -- Explore the nature of the work:
- "What types of work will you be doing? (code, research, documentation, analysis, etc.)"
- "How does your work typically evolve -- discrete chunks, continuous development, or mixed?"
- "Will you collaborate with others? How is work shared and integrated?"

**Current Practices** -- Assess existing habits:
- "How do you currently organize working files? What works well and what does not?"
- "What types of files do you typically create? How do they relate to each other?"
- "How do files change over time? How do you track changes?"

**Structure Development** -- Help design file organization:
- Identify major work categories based on the project's nature.
- Define relationships and dependencies between categories.
- Plan for how the structure will evolve as the project grows.
- Start with minimal organization; add structure only as needed.

**Documentation Standards** -- Establish per-file requirements:
- **Identity**: Each file states its purpose, role, and contribution to project objectives.
- **Relationships**: Each file documents its dependencies and connections to other components.
- **Evolution**: Each file tracks significant changes, rationale, and impact on other components.
- **Integration**: Each file maintains links to the Project Plan (which objectives it supports) and Living Questions (which questions/insights it addresses).

**Quality Standards** -- Define:
- Clarity and navigability expectations.
- Version tracking and change documentation conventions.
- Integration effectiveness checks.

## Component Dependencies

Use this map to understand how components inform each other during setup:

```
Project Foundations
  --> Collaborator Profile (foundations shape profile focus areas)
  --> Engagement Guidelines (foundations inform quality standards and methods)
  --> Project Plan (foundations define scope and objectives)
  --> Living Questions (foundations drive initial knowledge mapping)

Collaborator Profile
  --> Engagement Guidelines (profile shapes interaction patterns)
  --> Session Template (profile informs session structure preferences)

Engagement Guidelines
  --> Session Template (guidelines define session protocols)

Living Questions <--> Project Plan (bidirectional: questions inform planning, plan drives investigation priorities)

Session Template
  --> Working Files (session template defines how artifacts are referenced and tracked)
```

When setting up a component, reference already-completed components to ensure consistency. When a dependency has not yet been set up, note assumptions that will need validation when it is.
